# Cluster 6: Sequence labeling and structure

> **The story of this cluster in one sentence.** Cluster 5 gave us per-word vectors but Cluster 4's classifiers predict *one label per document* — many NLP tasks need *one label per token*, and the right label for *this* token depends on the labels for *neighbouring* tokens, so we need a sequence-aware model.

## Why this cluster exists

POS tagging ("DT NN VBD" for "the dog barked"), NER ("PERSON ORG LOCATION"), and dependency parsing all share a structural pattern: **each token in the sentence gets its own label, and the labels are not independent of each other.** Naïve Bayes per token would treat each in isolation; a vanilla LR would too. The conceptual move is to model the **transition probabilities between adjacent labels** — `P(next tag | current tag)` — and then **decode** the most probable label sequence given the observations. The hidden Markov model (HMM) is the cleanest formulation; the Viterbi algorithm is its dynamic-programming decoder.

The blueprint flags this as **very high weight**: **Exercise 3 of the mock is a 2-state HMM Viterbi by hand** (10 pts). MCQs cover HMM mechanics, the max-vs-sum trap, POS errors, NER span+type evaluation, and dependency parser output (mock Q9–Q11; Quiz III Q10–Q15, Q19–Q20 + Model B). The formula sheet provides the **complete Viterbi recursion** (init `δ₁(i)`, recursion `δₜ(j) = maxᵢ[…]`, backpointers, backtracking). What's *not* on the sheet: the *interpretation* — why **max** replaces **sum**, why backpointers exist, why span-and-type is required for NER credit.

## The arc

### 1. [[hidden-markov-model]] — the formal model behind sequence labeling

An HMM models a sequence of observations `w₁, …, wₙ` as generated by a parallel sequence of **hidden states** `t₁, …, tₙ`, with two assumptions:

1. **Markov assumption**: `P(tₜ | t₁ … tₜ₋₁) = P(tₜ | tₜ₋₁)` — the next state depends only on the current.
2. **Output independence**: `P(wₜ | …) = P(wₜ | tₜ)` — the observed word depends only on its tag.

For POS tagging: states = tags (NN, VB, DT, …), observations = words. The model has three parameter tables: **initial probabilities** `π_i`, **transition probabilities** `P(j | i)`, **emission probabilities** `P(w | t)`. All trained from a tagged corpus. HMM is *generative* — it models `P(tags, words)` jointly. Quiz III Q11, Q19 (and B) test the structural definitions.

### 2. [[hmm-viterbi]] — the dynamic-programming decoder

Given an HMM and an observation sequence, find the **most likely tag sequence** `argmax P(tags | words)`. Brute force is `|T|^n` (intractable for `n = 20`, `|T| = 12` → 4 × 10²¹ paths). **Viterbi** reduces this to `O(n · |T|²)` with a forward DP pass:

- **Init**: `δ₁(j) = π_j · P(w₁ | j)` for each tag `j`
- **Recursion**: `δₜ(j) = maxᵢ[δₜ₋₁(i) · P(j | i)] · P(wₜ | j)`
- **Backpointers** `ψₜ(j)`: store the argmax `i` from each recursion step
- **Termination + backtracking**: `t̂ₙ = argmax δₙ(j)`; walk backpointers backward to recover the path

The crucial conceptual move from the **forward algorithm** (which sums over previous states to compute `P(words)`) to **Viterbi** (which takes the max over previous states to find `argmax P(tags | words)`) is **swapping `Σ` for `max`**. That's why backpointers exist: max picks one predecessor, and you have to remember which to reconstruct the path. **Exercise 3 of the mock is a 2-state HMM by hand** — do the worked example in [[hmm-viterbi]] until you can fill the δ-table and backpointers without hesitation.

### 3. [[part-of-speech-tagging]] — the canonical HMM application

POS tagging assigns a syntactic category (NN, VB, DT, JJ, …) to each word in a sentence. It's the canonical application of HMM-style sequence labeling because (a) tags exhibit strong local order (DT followed by NN is common; DT followed by VB is rare), and (b) emission probabilities are well-estimated from corpora (most words have one or two POS tags with sharp distributions). Modern POS taggers use bidirectional LSTMs or transformers, but the conceptual model is still "states = tags, observations = words, decode via dynamic programming." Mock Q9 frames POS as the canonical HMM example.

### 4. [[named-entity-recognition]] — sequence labeling for spans

NER tags spans of tokens with entity types: `[Barack Obama]_PERSON visited [Paris]_LOCATION`. It's structurally similar to POS — a label per token — but with a key wrinkle: **labels are about spans**, not words. The standard encoding is **BIO** (Begin / Inside / Outside): "Barack" → B-PERSON, "Obama" → I-PERSON, "visited" → O, "Paris" → B-LOCATION. Evaluation requires **both the span boundaries and the entity type to be correct** — getting the span right with the wrong type counts as wrong (Quiz III Q15, Q20). This makes NER metrics noticeably stricter than POS metrics. Mock Q10 hits this directly.

### 5. [[dependency-parsing]] — structure beyond labels

Dependency parsing goes one step beyond per-token labeling: it produces a **tree of binary head-modifier relations** between words ("dog" is the subject of "barked"; "the" modifies "dog"). The output is a **rooted directed tree** over the tokens, with each edge labeled by a syntactic relation (nsubj, det, dobj, …). Dependency parsing is the modern alternative to CFG-based phrase-structure parsing (Cluster 1) — it's faster to compute, easier to learn, and provides directly usable input for downstream tasks (e.g. open-IE, machine translation). Mock Q11 tests *what the output looks like*: a tree of grammatical relations, not a sequence of tags. The exam weight is low here, but the concept is the bridge from "labels in a row" to "structure in a sentence."

## Connections worth seeing

- **Viterbi is the discrete-state cousin of the dot-product attention argmax in Cluster 8.** Viterbi: dynamic programming over discrete tag states, max over predecessors. Attention: continuous weighting over all positions, soft argmax via softmax. Both ask "given everything I've seen, what's the best label for *this* position?" — but Viterbi commits to a hard discrete path and attention computes a smooth weighted blend.
- **HMM emissions `P(w | t)` are the same shape as Naïve Bayes' `P(xᵢ | y)`** (Cluster 4). Both are "given the latent class, the observation is generated independently." NB is the *single-token* version; HMM strings together a *sequence* of NB emissions linked by transitions. *HMM is NB with memory.*
- **The max-vs-sum swap in Viterbi is a recurring template.** The **forward algorithm** sums `δₜ(j) = Σᵢ […]` to compute marginal likelihood; **Viterbi** takes max. Same DP table, different operator → different answer (likelihood vs. argmax path). This template — "compute the same recursion but change the reduction operator" — also shows up in beam search vs. exhaustive search (Cluster 9), where beam search keeps the top-k partial paths instead of just the max.
- **NER's BIO encoding is a clever workaround for HMM's "one tag per token" assumption.** NER fundamentally wants spans, but the model wants per-token labels. BIO bridges the two. Foreshadows how transformers in Cluster 9 handle QA — the answer is also a *span*, encoded by predicting *start* and *end* token indices independently.

## Common confusions

- **HMM forward vs. Viterbi** — same DP table; forward sums (computes likelihood); Viterbi maxes (computes best path).
- **POS vs. NER** — POS tags are syntactic categories (every word gets one); NER tags are entity types over *spans* (most words are O = "Outside"). NER evaluation requires span+type, not just type.
- **Per-word HMM probability** — the contribution at each step is `P(tag | prev_tag) · P(word | tag)` — *multiplied*, not summed.
- **Brute force vs. Viterbi cost** — brute is `|T|^n`; Viterbi is `O(n · |T|²)`. The savings are exponential because DP reuses partial paths.
- **Dependency parsing output** — a tree of *grammatical relations between words*, not a sequence of tags.

2. **HMM is generative; logistic regression for sequence labeling (LR-per-token + transition features → CRF) is discriminative.** What does each model's strengths suggest about *when you'd choose which*? Tie back to Cluster 4's generative-vs-discriminative trade-off.
3. **For a 3-word sentence "they can fish" with 2 possible tags {N, V}**, sketch the δ-table layout (rows = tags, columns = words, plus backpointers). You don't need actual numbers — just demonstrate you can lay out the structure under exam pressure.
4. **NER credit requires both correct span and correct type.** Construct a hypothetical NER prediction that gets *high token-level accuracy* but *low entity-level F1*. What does this say about choosing the right granularity for evaluation?
5. **Dependency parsing produces a tree, POS tagging produces a sequence.** Which is structurally richer? Which is more useful as input to downstream modules like attention (Cluster 8) — and why?
6. **Looking back to Cluster 4:** HMM's emission `P(w | t)` is the same shape as NB's `P(xᵢ | y)`. Why doesn't NB-per-token work for POS tagging? What does HMM add that NB lacks?
7. **Looking forward to Cluster 7:** HMM treats tag transitions as a fixed-table lookup (`P(j | i)` is `|T|²` learned numbers). What if the "transition probability" depended on *more than just the previous tag* — say, the entire prefix of the sentence? What kind of model would you need?
